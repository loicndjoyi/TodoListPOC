# Architecture Decision Records (ADRs)

> These records capture the key architectural decisions made for this project and the reasoning behind them.  
> Format: Michael Nygard's ADR template

---

## ADR-001: Service Layer over CQRS/MediatR

**Status**: Accepted

### Context
We need to structure the backend business logic for a simple CRUD API (4 endpoints, 1 entity). MediatR with CQRS is commonly used in modern .NET to decouple controllers from logic, but this is a time-boxed Proof of Concept (POC).

### Decision
Use a simple **Service Layer** (`ITaskService` → `TaskService`) injected via Dependency Injection.

### Consequences
- ✅ Respects the **SRP** and **DIP** (SOLID principles) without unnecessary ceremony.
- ✅ Faster to implement and easier to parse for code reviewers.
- ✅ Fully testable via interface mocking (NSubstitute).
- ⚠️ Risks becoming a "God Service" if the application scales drastically (would require a refactor to MediatR).

## ADR-002: SQLite as Data Store

**Status**: Accepted

### Context
We need a storage solution that is entirely portable, requires zero infrastructure setup, yet demonstrates relational database (SQL) integration skills.

### Decision
Use **SQLite** paired with **Entity Framework Core** (Code-First approach, embedded migrations).

### Consequences
- ✅ Zero server configuration required (no Docker needed to run the project).
- ✅ Real SQL structure — migrating to SQL Server later requires only a connection string change.
- ✅ Embedded Migrations folder demonstrates mastery of the database lifecycle.
- ⚠️ No native robust optimistic concurrency (`rowversion`). Acceptable for a local POC.
- ⚠️ Potential file-locking issues (CIFS) if deployed to Linux-based Azure App Services.

## ADR-003: Global Exception Handler (ProblemDetails)

**Status**: Accepted

### Context
Enterprise APIs must return standardized, machine-readable error responses. .NET 8 introduced `IExceptionHandler` as the recommended approach to replace custom error middlewares.

### Decision
Implement `IExceptionHandler` to intercept all unhandled exceptions and expose them using the **RFC 7807 ProblemDetails** JSON format.

### Consequences
- ✅ Predictable and consistent error response format for the frontend.
- ✅ Safe for production: stack traces and sensitive error details are not leaked.
- ✅ Demonstrates an industry-standard engineering practice.

## ADR-004: Angular 21 Standalone Components & Signals

**Status**: Accepted

### Context
The frontend needs to remain efficient to build while demonstrating proficiency with modern web frameworks.

### Decision
Use **Angular 21** featuring:
- **Standalone Components** (avoiding heavy NgModules)
- **Signals** for reactive local state management
- **Reactive Forms** for structured and testable inputs
- **HTTP Interceptor** for centralized API handling

### Consequences
- ✅ Illustrates up-to-date knowledge of the 2024-2026 Angular paradigms.
- ✅ Signals drastically reduce the verbosity of RxJS for simple state.
- ✅ Reactive Forms allow for easier logic extraction and unit testing.

---

## ADR-005: Automated Continuous Integration (CI)

**Status**: Accepted

### Context
Code building and testing should not rely on manual verification, even for a POC.

### Decision
Integrate CI workflows using **GitHub Actions** (`.github/workflows/ci.yml`).

### Consequences
- ✅ Quality gates are enforced on every push or Pull Request against the main branch.
- ✅ Proves a baseline understanding of DevOps and release automation.
